# Rover — Журнал решений (DECISIONS)

Журнал архитектурных и технических решений. Каждая запись: что решили, какие альтернативы рассматривали, почему выбрали именно так.

---

## SB-000. Правила ведения документации

**Решение:**

- `SPEC.md` и `DECISIONS.md` — общие для всего монорепо, лежат в корне.
- `AGENT.md` и `README.md` — свои в каждой подпапке (`back/`, `app/`, `card/`).
- При изменении логики, модели данных или контрактов **во время разработки или тестирования** — синхронно обновляются `SPEC.md` (что и как) и `DECISIONS.md` (почему именно так).
- `SPEC.md` и `DECISIONS.md` как правило меняются вместе: SPEC отражает что должно быть, DECISIONS объясняет, почему пришли к этому варианту.
- `AGENT.md` обновляется:
  - После каждого повышения минорной версии (`x.y.0` → `x.(y+1).0`).
  - При появлении важных операционных нюансов (новая команда, изменение структуры, новые правила работы агента).

**Обоснование:**
- Общий SPEC исключает рассинхрон между бэком, фронтом и карточкой — они говорят на одном языке.
- Парность SPEC ↔ DECISIONS даёт документации память: можно посмотреть не только «как должно работать», но и «почему мы отказались от альтернативы X».
- AGENT редко меняющийся документ — обновлять его в каждом патче было бы шумом.

## SB-001. Монорепо вместо нескольких репозиториев

**Решение:** один репозиторий `Rover` с подпапками `back/`, `app/`, `card/`. Общие документы (`SPEC.md`, `DECISIONS.md`, корневой `README.md`) — в корне.

**Обоснование:**
- Один источник истины по протоколу: невозможно «забыть» обновить SPEC в одном из трёх репо.
- Один issue tracker — баги протокола, бэка, фронта и карточки в одном месте.
- Атомарные коммиты: изменение протокола = синхронный коммит, меняющий и SPEC, и три реализации.
- Цена: версии трёх компонентов идут независимо (см. SB-002), не «релизим всё одной командой».

---

## SB-002. Независимые версии у back / app / card

**Решение:** каждый компонент имеет собственную версию. Git-теги вида `back-1.0.0`, `app-1.0.0`, `card-1.0.0`. Изменение протокола, ломающее совместимость, = синхронный bump всех трёх + обновление SPEC.

**Обоснование:**
- Карточка может обновляться чаще приложения — UI правки не должны принуждать к релизу APK.
- Бэк может фиксить баг HA-интеграции без перерелиза остальных.
- Совместимость гарантируется через SPEC, а не через монолитную версию.

## SB-003. Транспорт — Meshtastic поверх LoRa

**Решение:** строим протокол поверх Meshtastic, порт `PRIVATE_APP = 256`.

**Обоснование:**
- Целевая среда — дом на колёсах в местах без интернета или с блокировками. Нужна радиосвязь, не зависящая от инфраструктуры.
- Meshtastic уже решает на уровне ниже: модуляция LoRa, mesh-маршрутизацию, базовое шифрование канала, duty cycle. Не переизобретаем.
- Альтернатива «голый LoRa без Meshtastic» отвергнута: пришлось бы писать MAC/маршрутизацию с нуля, теряя совместимость с экосистемой готовых нод.

## SB-004. Сериализация — MessagePack

**Решение:** все пакеты сериализуются MessagePack.

**Обоснование:**
- Бинарный формат — экономия по сравнению с JSON критична: пакет LoRa ограничен ~230 байтами payload, накладные расходы JSON съели бы половину.
- Поддержка в Python (бэк), Kotlin (фронт) и JavaScript (карточка) — зрелая.
- Альтернатива Protobuf отвергнута: требует синхронных схем, лишняя жёсткость для MVP, где поля ещё меняются.

**Следствие:** числовые поля могут прийти как Int / Long / UInt в зависимости от размера значения. Все стороны обязаны это принимать.

---

## SB-005. Все пакеты — broadcast

**Решение:** `to = 0xFFFFFFFF` для всех пакетов в обе стороны.

**Обоснование:**
- Естественно работает для нескольких фронтов: один PUSH достигает всех телефонов.
- Не требует регистрации node ID каждого фронта на бэке.
- Meshtastic для broadcast на канале не делает дополнительных подтверждений — экономим эфир.
- Цена: чужие пакеты (если канал общий) проходят парсер. Решается приватным каналом — см. SB-014.

## SB-006. Отказ от CONFIRM

**Решение:** убран тип пакета `tp=1` (CONFIRM). Подтверждение исполнения команды — обычный PUSH с новым состоянием.

**Обоснование:**
- В мире с несколькими фронтами CONFIRM создавал бы дублирование: бэк всё равно должен слать PUSH остальным фронтам, чтобы они увидели изменение. CONFIRM был отдельным пакетом, идущим инициатору, плюс PUSH для остальных — лишний эфир.
- PUSH самодостаточен: содержит реальное состояние, прочитанное из HA после команды. Для инициатора это и подтверждение, и обновление; для остальных — только обновление.
- Цена: нет явного маркера «команда дошла». Если PUSH потерялся, фронт не различает «бэк не получил CMD» и «бэк выполнил, но PUSH дропнулся». На MVP это приемлемо.

**Альтернатива:** оставить CONFIRM для критичных команд (alarm arming, lock). Отложено, см. открытые вопросы спецификации.

## SB-007. Подтверждение через бесшумные повторы, без сообщения об ошибке

**Решение:** если PUSH не пришёл — фронт молча повторяет CMD заданное число раз (по умолчанию 5) с периодом (по умолчанию 15 сек). Ошибку пользователю не показываем.

**Обоснование:**
- В условиях нестабильной радиосвязи временные потери — норма. Показывать ошибку каждый раз — спам и тревога.
- Устройство остаётся «серым» в UI до получения PUSH — это и есть индикация неподтверждённости.
- Пользователь сам решит, повторить ли руками, увидев, что ничего не изменилось.

## SB-008. Очередь — Map по `short_id`, не FIFO/LIFO

**Решение:** ключ исходящей очереди = `short_id` устройства. Новая команда для того же устройства затирает старую.

**Обоснование:**
- В UI пользователь может быстро кликнуть туда-сюда («передумал»). Промежуточные команды бессмысленно гонять по эфиру — важно только финальное намерение.
- LIFO/FIFO не подходит: при FIFO повторно поставленная команда уходила бы в конец и выполнялась с задержкой; при LIFO старая команда всё равно где-то лежит и тратит память.
- Затирание = «последнее намерение пользователя побеждает», что соответствует ожиданиям UX.

**Следствие:** счётчик повторов при затирании сбрасывается. Новая команда — новый цикл.

## SB-009. Сортировка очереди по `next_send_time`

**Решение:** очередь сортируется по времени следующей отправки. Новые сообщения с `next_send_time = now` оказываются вверху и отправляются немедленно; отправленные уходят в конец с `next_send_time = now + период`.

**Обоснование:**
- Просто: одна структура, одна операция сортировки, не нужны отдельные «активная очередь» и «очередь ретраев».
- Round-robin между сообщениями: ни одно не блокируется другим, повторы естественно чередуются с новыми.
- Не требует приоритетов — все сообщения равноправны, что упрощает мышление.

## SB-010. Пачки до 200 байт, фрагментация только для крупных сообщений

**Решение:** процедура отправки собирает в один пакет столько сообщений, сколько влезает. Сообщения, не помещающиеся в один пакет даже без соседей, фрагментируются через конверт `tp=7`.

**Обоснование:**
- Меньше пакетов в эфире = меньше нагрузка на duty cycle и меньше конкуренция за канал.
- Фрагментация выделена в отдельный слой с собственным конвертом, а не размазана по типам пакетов: парсер фрагментов не знает про CMD/PUSH/STATUS, склеивает байты и передаёт дальше.
- Цена: дополнительный тип пакета, дополнительная логика буфера на приёмнике. Окупается тем, что большинство пакетов (CMD, PUSH) фрагментировать не нужно — это редкий случай для крупных секций CONFIG.

**Отвергнутая альтернатива:** добавить поля `fid/fi/fn` в каждый тип пакета. Замусоривает протокол, фрагментация перестаёт быть прозрачным слоем.

## SB-011. Ретраи фрагментов не предусмотрены

**Решение:** если фрагмент потерялся и сборка не завершилась за 60 сек — буфер дропается, отдельных ретраев фрагментов нет.

**Обоснование:**
- Фрагментируются на практике только пакеты CONFIG. У них есть естественный механизм повторного запроса: фронт видит расхождение `cfgh` и заново шлёт `cfg=2`.
- Реализация ретраев на уровне фрагментов потребовала бы NACK-ов и состояния — слишком сложно для выигрыша, который полностью покрывает верхний слой.

## SB-012. `short_id` — 2 байта Int, фиксированный размер

**Решение:** идентификатор устройства — 16-битный Int, передаётся как число (не строка).

**Обоснование:**
- 2 байта = 65536 значений. С запасом для любого реалистичного дома, коллизии редки.
- Int короче строкового представления в MessagePack: ~3 байта против ~5–6. На пачках экономия складывается.
- Фиксированная длина проще: парсеру не нужно догадываться, где кончается ID. Переменная длина (3/4/5 hex-знаков в зависимости от числа устройств) даёт мизерный выигрыш и добавляет сложности.

**Коллизии:** при добавлении нового устройства, если первые 2 байта MD5 от `entity_id` уже заняты, делаем повторный MD5 с солтом до получения уникального значения. Назначенный ID за устройством закреплён навсегда.

## SB-013. Хранение обработанных сообщений 15 минут

**Решение:** обработанные / устаревшие сообщения не удаляются из памяти мгновенно, а помечаются и хранятся 15 минут для дедупликации.

**Обоснование:**
- Meshtastic mesh может ретранслировать пакеты — дубль одного и того же сообщения может прилететь повторно через 1–10 секунд.
- 15 минут — с запасом покрывает реалистичные задержки ретрансляции и часовых поясов между нодами.
- Альтернатива «удалять мгновенно» опасна: дубль CMD выполнится повторно (переключит свет туда-обратно).

## SB-014. PING / PONG — простой keepalive, без cfgh

**Решение:** PING / PONG несут только timestamp для эха. Хеш конфига передаётся отдельно в META.

**Обоснование:**
- Раннее было предложение пихать `cfgh` в каждый PONG — фронт автоматически сверял бы. Отвергнуто: смешивает функции, утяжеляет каждый keepalive, навязывает фронту логику синхронизации.
- Простой PING/PONG = одна простая задача. Когда фронт хочет проверить актуальность конфига — он сам запрашивает META.
- META — это маленький пакет, спросить его недорого.

## SB-015. Приватный канал Meshtastic для production

**Решение:** на MVP допустим LongFast / `AQ==`. В production — обязательно приватный канал с собственным именем и PSK.

**Обоснование:**
- LongFast — общая «комната» Meshtastic-пользователей. Чужие пакеты грузят парсер, наши пакеты грузят им эфир.
- Приватный канал = логическая изоляция: пакеты других каналов отбрасываются по хешу до парсинга, не доходят до приложения.
- Физически одна частота, эфир общий, но контента друг друга не видим.
- Несколько фронтов работают на одном приватном канале — broadcast достигает всех «своих».

**Цена:** дополнительный шаг при онбординге (импорт ключа канала на телефон). На MVP — вручную, в production — через QR.

## SB-016. Hop limit по умолчанию 0

**Решение:** hop limit настраивается в META, дефолт `0` (прямая связь).

**Обоснование:**
- В типичной схеме RV: один шлюз + телефоны в радиусе видимости. Ретранслировать некому — hop>0 бессмыслен.
- Чужие ноды Meshtastic не на нашем приватном канале — наши пакеты не ретранслируют.
- Hop>0 полезен, только если развернуты собственные ретрансляторы для расширения зоны вокруг RV. Тогда — увеличить в конфиге.

## SB-017. Несколько фронтов — допустимы, без явной регистрации

**Решение:** к одному шлюзу могут подключаться несколько экземпляров приложения. Бэк не различает фронты по node ID, не ведёт реестра.

**Обоснование:**
- Broadcast-модель уже это поддерживает «бесплатно» — каждый PUSH видят все.
- Регистрация фронтов добавила бы состояние, которое нужно поддерживать и которое усложняет логику оффлайна.
- Цена: если два пользователя одновременно дают команду одному устройству — последняя выигрывает, без явных конфликтов. На практике это редко и приемлемо.

---

## SB-018. Алерты повторяются раз в 15 секунд, пока активны

**Решение:** активные алерты (binary_sensor определённого класса в состоянии on) бэк шлёт PUSH-ом каждые 15 сек. При снятии — финальный PUSH и остановка.

**Обоснование:**
- Безусловная гарантия доставки нужна именно для алертов: пропустить открытую дверь / газ / задымление — критично.
- 15 сек — компромисс между «не пропустить» и «не забить эфир».
- Повторы группируются в одну пачку (см. SB-010), если активных алертов несколько.

---

## SB-019. Бэк не копит очередь, если фронты оффлайн

**Решение:** при отсутствии фронтов (не приходит PING > 10 минут) бэк не накапливает PUSH и алерты в очереди.

**Обоснование:**
- Когда фронт вернётся, он сам запросит `req=all` и получит актуальные состояния. История изменений за период оффлайна не нужна.
- Алерты, если всё ещё активны на момент возвращения фронта, будут отправлены в следующем 15-секундном цикле — не пропадут.
- Накопление очереди в оффлайне создало бы лавину при возвращении и неограниченный рост памяти.

## SB-020. Логи — на всех трёх компонентах

**Решение:** ведём логи на бэке (файлы, ротация), приложении (кольцевой буфер + экспорт) и карточке (буфер в памяти браузера).

**Обоснование:**
- Радиосеть капризна: без логов отладка проблем «у меня выключатель сегодня не сработал» превращается в гадание.
- Бэк ближе к серверу — там полноценная файловая ротация.
- Приложение ограничено по памяти и хранилищу — кольцевой буфер в памяти + ручной экспорт по нажатию (отправить разработчику).
- Карточка живёт в браузере — встроенные DevTools плюс собственный буфер, который пишем сами.

---

## SB-021. PIN-коды и пароли — только как хеши

**Решение:** PIN от сигнализации, любые секреты передаются как SHA-256 хеши с общей солью.

**Обоснование:**
- Даже на приватном канале Meshtastic ключ канала может утечь. Дополнительный слой защиты для критичных данных не помешает.
- Бэк на стороне HA знает соль и валидирует хеш — pin в открытом виде по радио не летает.

---

## SB-022. На MVP — конфиг копируется вручную, без QR-онбординга

**Решение:** на этапе MVP пользователь вручную вводит / копирует параметры подключения (имя канала, PSK, node ID шлюза). QR-онбординг — после MVP, реализуется в карточке.

**Обоснование:**
- MVP должен быть работающим минимумом. QR — UX-улучшение, не функциональность.
- Реализация QR-онбординга требует продумывания формата (что именно в QR), безопасности (защита от подмены), UI на обеих сторонах. Откладываем.
- Карточка — естественное место для отображения QR (на экране HA, доступном админу).

---

## SB-023. Стек для card — TypeScript + LitElement + Rollup

**Решение:** карточка Lovelace пишется на TypeScript с LitElement, сборка — Rollup.

**Обоснование:**
- Стандарт де-факто для кастомных карточек HA: документация, примеры, типы для `HomeAssistant` объекта — всё под это заточено.
- TypeScript критичен при работе с маленькой моделью (см. SB-024) — типизация ловит ошибки в форматах данных HA, которые модель иначе пропустит.
- Rollup — самый распространённый бандлер для Lovelace-карточек, opencode скорее найдёт правильные паттерны.

## SB-024. Разработка через opencode + промпты, цикл версионирования

**Решение:**

- Промпты для opencode (модель qwen 3.6 35b a3b distilled opus 4.7) пишутся узкими, конкретными, с явными ограничениями.
- Цикл: промпт → код → коммит с патчем `x.y.1` → тест → если баги, новый промпт → патч `x.y.2`, `x.y.3`, … → когда всё работает, итоговый тест → коммит с минорным повышением `x.(y+1).0`.
- Принципы промптов: один промпт = одна задача, явные имена файлов/функций/сигнатур, запрет на изменения вне указанной области, минимальные примеры на вход/выход.

**Обоснование:**
- Маленькая модель плохо угадывает контекст — промпт должен исключать неоднозначность.
- Шаг с патчами даёт точку отката после каждой неудачной попытки и историю проб.
- Минорное повышение версии = подтверждённая рабочая итерация.

## SB-025. Релизы с ZIP для HACS и карточки

**Решение:** при каждом минорном (а где требуется — и патчевом) повышении версии для компонентов, устанавливаемых через HACS (`back/`) и фронтенд-карточки (`card/`) — создаётся GitHub Release с прикреплённым ZIP-архивом.

**Обоснование:**
- HACS требует ZIP-релизы для версионированной установки интеграций и карточек.
- Без релизов пользователи получают только git-снимок main, что не годится для стабильных версий.
- Автоматизация: GitHub Actions собирает ZIP при создании тега `back-x.y.z` / `card-x.y.z` и публикует Release.

## SB-026. Карточка card — мониторинг + админ, без дублирования управления устройствами

**Решение:** card отображает статус Rover (имя дома, фронты, поток пакетов) и предоставляет админ-функции (QR онбординга по паролю). Управление устройствами в карточке не реализуется.

**Обоснование:**
- Устройства уже есть в HA-нативно — дублировать переключатели в Rover-карточке бессмысленно.
- Карточка решает уникальную задачу: показать, что происходит **в Rover-радиосети**, чего нельзя увидеть стандартными средствами HA.
- Админ-функции (QR) логично размещать там же, где админ и так работает — на дашборде HA.

## SB-027. Фрагментация только в пустой пачке

**Решение:** при формировании пачки Batcher применяет фрагментацию только когда текущая пачка пуста. Если большое сообщение встретилось при уже непустой пачке — оно пропускается в этом цикле и попробует уехать в следующем, когда станет первым.

Алгоритм Batcher:
1. Идём по очереди от верха.
2. Для каждого готового к отправке сообщения смотрим размер `encode(msg)`.
3. Если **пачка пуста и сообщение > MAX_PACKET_SIZE** — фрагментируем через `codec.fragment()`, пачка становится серией tp=7, закрывается. Других сообщений не добавляем.
4. Если **сообщение влезает в остаток пачки** — добавляем.
5. Если **не влезает** — пропускаем (next_send_time не трогаем), идём к следующему. Может что-то мелкое ещё влезет.

**Обоснование:**
- Фрагментация — дорогая операция (несколько пакетов в эфире). Смешивать фрагменты с другими сообщениями в одной пачке нет смысла: всё равно пакетов будет несколько.
- Дать большому сообщению дождаться пустой пачки = простой и предсказуемый алгоритм без специальных случаев «фрагмент плюс хвост из мелких».
- Цена: большое сообщение может задержаться на один цикл очереди (≤ 15 сек по умолчанию). Для CONFIG это приемлемо.

## SB-028. Выделены чистые модули commands.py и state_extractor.py

**Решение:** дополнительно к 11 файлам бэка из изначального плана добавляются два чистых модуля:

- **`commands.py`** — преобразование `(Device, packet) → ServiceCall(domain, service, data)`. Возвращает описание вызова HA-сервиса, но сам ничего не вызывает.
- **`state_extractor.py`** — преобразование `(Device, HA state) → dict состояния протокола`. Чтение без побочных эффектов.

`handlers.py` использует их как библиотеки преобразований. `ha_bridge.py` остаётся тонкой обёрткой над реальными вызовами HA.

**Обоснование:**
- В референсной реализации логика «по типу устройства определить, какой сервис HA вызвать с какими параметрами» занимала ~350 строк switch-case. Оставлять её внутри handlers.py смешивает оркестрацию (что сделать в каком порядке) и преобразование (как именно).
- Чистые модули тестируются без моков HA: даёшь Device + packet, проверяешь возвращённую структуру.
- Граница между «знанием о протоколе HA» (commands/state_extractor) и «выполнением» (ha_bridge) делает интеграционные тесты с реальным HA минимальными — большая часть логики покрывается юнит-тестами.

**Цена:** два дополнительных файла. Совокупная сложность та же, но разнесена по слоям.

## SB-029. cfgh считается по экспортируемому виду конфига

**Решение:** хеш конфига `cfgh` вычисляется только по тем данным, которые реально уходят на фронт через CONFIG-пакеты — без `entity_id` и других полей, которые видны только бэку.

```
cfgh = MD5(sort_keys_json({
    devices: export_devices(),   # {short_id, n, t, a}
    areas:   export_areas(),     # {id, n}
    users:   export_users()      # {id, hash}
}))[:8]
```

**Обоснование:**
- Фронт сравнивает `cfgh` именно для того, чтобы понять «нужно ли мне перезапросить CONFIG, есть ли новости». Если cfgh меняется из-за того, что у бэка поменялся внутренний `entity_id` (а на фронт всё равно уходит то же самое) — это «фантомное» расхождение, оно вызовет бесполезный перезапрос конфига.
- Хешируя только экспортируемый вид, мы гарантируем: cfgh меняется тогда и только тогда, когда у фронта реально устарели данные.
- Альтернатива (хешировать всё подряд на бэке) была отвергнута: создаёт ложные срабатывания, тратит эфир и батарею фронта.

**Источник истины:**
- Бэк хранит полные записи: `{short_id, entity_id, t, n, a, u}` для устройств, плюс areas и users целиком.
- Фронт долговременно не хранит ничего, кроме параметров подключения и кеша последнего полученного конфига (для оффлайн-отображения).
- При запуске фронт пингует, при необходимости запрашивает META, сравнивает cfgh с локальным, при расхождении — запрашивает полный CONFIG.

## SB-030. Light brightness в протоколе — 0..100 (проценты)

**Решение:** поле `bri` в пакетах CMD/PUSH/STATUS — целое 0..100. Бэк конвертирует в HA brightness (0..255) при вызове сервиса.

**Обоснование:**
- Один байт в MessagePack для значений 0..127 — экономия по сравнению с 0..255 (там тоже один байт, но семантически меньше понимания у пользователя/UI).
- Унификация с другими «процентными» полями: `sp` (fan speed) и `pos` (cover position) тоже идут в 0..100.
- Точность 1% более чем достаточна — глаз человека не различает шаги яркости меньше 2-3%.
- Конверсия в HA-формат — на стороне бэка, в `commands.py`, в одном месте.

## SB-031. Alarm cmd — фиксированный список значений

**Решение:** поле `cmd` для типа устройства `A` (alarm_control_panel) принимает ровно одно из значений: `arm_home`, `arm_away`, `arm_night`, `disarm`. Поле `pin` (если указано) передаётся как параметр `code` в HA-сервис.

**Обоснование:**
- HA имеет соответствующие сервисы `alarm_arm_home`, `alarm_arm_away`, `alarm_arm_night`, `alarm_disarm` — наш набор симметричен.
- Жёсткий список = валидация на стороне бэка простая: неизвестный `cmd` отвергается без вызова сервиса.
- Других режимов armed (например, `arm_vacation`, `arm_custom_bypass`) в HA существуют, но в MVP их не поддерживаем — добавим, если возникнет реальная нужда.

## SB-032. Источник истины — бэк; фронт ничего не хранит долговременно

**Решение:** все данные конфига (устройства, зоны, пользователи) хранятся на бэке. Фронт долговременно хранит только параметры подключения (адрес шлюза, ключ канала, период повторов, число попыток, hop limit) и **кеш последнего полученного CONFIG** — для отображения устройств в оффлайне.

**Обоснование:**
- Единый источник истины — простая модель: «что на бэке, то и правда». Никакой синхронизации, никаких конфликтов.
- При смене устройства/зоны/пользователя пользователь правит в HA (или в админ-карточке) — бэк пересчитывает cfgh — фронт при следующем пинге увидит расхождение → запросит обновление.
- Фронт может работать в оффлайне с устаревшими данными — это лучше, чем чёрный экран.
- Авторизация на фронте локальная (хеши паролей пользователей приходят в составе CONFIG) — это позволяет не зависеть от связи в момент входа.
