# Rover — Спецификация протокола и системы

**Версия:** MVP 0.1
**Статус:** черновик после согласования

---

## 1. Назначение системы

Rover (**R**emote **O**ver **R**adio) — система дистанционного управления умным домом Home Assistant в условиях отсутствия интернета или его блокировки. Целевая среда — дом на колёсах (RV), яхта, дача. Связь между мобильным приложением и шлюзом обеспечивается радиосетью LoRa через стек Meshtastic.

## 2. Архитектура

Система состоит из трёх компонентов:

- **back** — кастомная интеграция Home Assistant (Python). Подключена к локальной Meshtastic-ноде (шлюзу). Слушает события устройств HA, принимает команды от приложения, ретранслирует изменения состояния.
- **app** — мобильное приложение под Android (Kotlin). Подключено к собственной Meshtastic-ноде (телефонная нода, BLE). Отображает состояние устройств, отправляет команды.
- **card** — карточка Lovelace для дашборда Home Assistant (TypeScript + LitElement). Показывает статус системы, подключённые фронты, поток пакетов. Доступ к админ-функциям (онбординг) — по паролю.

Допускается одновременная работа нескольких экземпляров приложения (нескольких телефонов) с одним шлюзом.

## 3. Транспортный слой

### 3.1 Радиосеть

- **Стек:** Meshtastic.
- **Канал:** приватный канал Meshtastic с заданным именем и PSK. На MVP допускается канал LongFast с дефолтным ключом `AQ==`, в production — обязательно приватный канал.
- **Порт приложения:** `PRIVATE_APP = 256`.
- **Hop limit:** настраивается, по умолчанию `0` (прямая связь шлюз ↔ телефон). Увеличивается при наличии своих ретрансляторов.
- **Адресация:** все пакеты отправляются broadcast (`to = 0xFFFFFFFF`). Приёмник принимает пакеты от любого node ID на своём канале.

### 3.2 Сериализация

- **Формат:** MessagePack (бинарный).
- **Лимит размера пакета:** 200 байт payload (с учётом MessagePack-конверта).
- **Обязательное поле во всех пакетах:** `tp` (Int) — тип пакета.

### 3.3 Совместимость типов

MessagePack может кодировать числа как разные внутренние типы (Int / Long / UInt). Приёмники **обязаны** принимать оба варианта для всех числовых полей. Сериализация на стороне отправителя должна выбирать минимальный возможный тип.

## 4. Типы пакетов

| tp | Направление | Название | Описание |
|----|-------------|----------|----------|
| 2 | HA → App | STATUS | Состояние устройства в ответ на запрос |
| 3 | HA → App | PUSH | Изменение состояния (инициативное или после команды) |
| 4 | HA → App | CONFIG | Конфигурация (META или секции с пагинацией) |
| 5 | App → HA | CMD | Команда или запрос |
| 6 | App ↔ HA | PING / PONG | Keepalive |
| 7 | Любое | FRAGMENT | Часть фрагментированного сообщения |

**Тип `tp=1` (CONFIRM) изъят из протокола.** Подтверждение исполнения команды заменено на broadcast PUSH с новым состоянием устройства (см. §6.2).

## 5. Идентификация устройств и хранение конфига

### 5.1 Источник истины

Все данные конфигурации (устройства, зоны, пользователи) хранятся **на бэке**. Фронт ничего не хранит долговременно, кроме параметров подключения и кеша последнего полученного конфига для работы при потере связи.

### 5.2 Хеш устройства

Каждое зарегистрированное в Rover устройство Home Assistant получает короткий идентификатор `short_id` — **2-байтовый Int** (16 бит, диапазон 0–65535), вычисляемый как первые 2 байта MD5 от `entity_id`.

При коллизии с уже зарегистрированным устройством — пересчёт с инкрементом солта до получения уникального значения. Назначенный однажды `short_id` за устройством закреплён и не меняется, даже если позже будет удалено конфликтовавшее устройство.

### 5.3 Что хранит бэк

Полная запись по каждому устройству в `Registry`:

```
short_id → {
    entity_id: "light.salon_main",   # HA entity_id (только локально, не уходит на фронт)
    t: "L",                          # тип устройства (см. §5.5)
    n: "Люстра",                     # friendly name из HA
    a: "salon",                      # area ID из HA Areas
    u: null                          # unit_of_measurement (для сенсоров)
}
```

Зоны (`areas`) — `{id, n}`, где `id` — идентификатор зоны из HA Areas, `n` — отображаемое имя.

Пользователи (`users`) — `{id, hash}`, где `id` — идентификатор пользователя, `hash` — SHA-256 пароля с солью. Пароли в открытом виде на бэке не хранятся.

### 5.4 Что уходит на фронт

Через CONFIG-пакеты передаётся **только то, что нужно фронту**:

- **devices** — `{short_id, n, t, a}` без `entity_id` (фронт не знает HA-структуру).
- **areas** — `{id, n}` целиком.
- **users** — `{id, hash}` для локальной авторизации на фронте.

В обычных пакетах (CMD, PUSH, STATUS) передаётся только `short_id` устройства — этого достаточно, бэк по `short_id` найдёт `entity_id` через свой Registry.

### 5.5 Типы устройств

| Код | HA domain | Команды | Поля состояния |
|-----|-----------|---------|----------------|
| L | light | s, bri, ct | s, bri, ct |
| SW | switch | s | s |
| C | climate | s, th, md, fn | s, th, tc, md, fn |
| WH | water_heater | s, th | s, th, tc, md |
| F | fan | s, sp | s, sp |
| CV | cover | cmd (open/close/stop), pos | st, pos |
| LK | lock | cmd (lock/unlock) | st |
| BS | binary_sensor | — | s |
| S | sensor | — | v (string), u |
| SI | siren | s | s |
| B | button / scene | cmd (press) | — |
| A | alarm_control_panel | cmd (arm_home/arm_away/arm_night/disarm), pin | st |
| H | humidifier | s, th | s, th, tc |

**Примечание про `bri` (яркость light):** диапазон `0..100` (проценты), не `0..255`. Бэк конвертирует в HA brightness при вызове сервиса (`HA_brightness = bri * 2.55`).

**Примечание про `th` для humidifier:** для типа `H` поле `th` означает целевую влажность (`humidity`), а не температуру.

**Примечание про `v` (sensor value):** передаётся как **строка**, а не Double. Это позволяет передавать любые значения HA: числа (`"23.5"`), enum-состояния (`"running"`), timestamp (`"2026-05-31T14:32:00+00:00"`). Парсинг и интерпретация — на стороне фронта.

## 6. Семантика обмена

### 6.1 Keepalive (PING / PONG)

- Фронт отправляет **PING** раз в 2 минуты.
- Бэк отвечает **PONG** на каждый PING.
- Если бэк не получает ни одного PING от фронта более **10 минут**, фронт считается оффлайн.
- Поле `ts` в PING — Unix timestamp отправителя, эхо в PONG. Никакой логики на расхождении времени не строится.

### 6.2 Жизненный цикл команды

1. Пользователь меняет значение на фронте (например, переключает свет). UI отображает устройство в «неподтверждённом» состоянии (серый).
2. Фронт ставит CMD в очередь отправки, ключ очереди = `short_id` устройства. Если в очереди уже была команда для этого устройства — она затирается, счётчик повторов сбрасывается.
3. Очередь отправляет пачку пакетов (см. §7).
4. Бэк получает CMD, вызывает соответствующий сервис Home Assistant, ждёт **500 мс**, читает актуальное состояние устройства из HA, отправляет broadcast **PUSH** с новым состоянием.
5. Все фронты получают PUSH. У инициатора — устройство выходит из серого состояния, новое значение применяется. У остальных фронтов — это просто обновление состояния.
6. Если PUSH не пришёл за `период_повтора × число_повторов`, на фронте устройство остаётся серым. Пользователь повторит вручную при необходимости. Сообщение об ошибке не показывается.

CONFIRM как отдельный тип пакета не используется. Подтверждение и инициативное обновление — один и тот же PUSH (`tp=3`).

### 6.3 Запрос актуальных состояний

- `req="all"` в CMD — бэк отвечает серией STATUS-пакетов (`tp=2`) по одному устройству. Пауза между пакетами регулируется duty cycle (см. §7).
- `req=<short_id>` — бэк отвечает одним STATUS по конкретному устройству.

### 6.4 Запрос конфигурации

- `cfg=1` в CMD — бэк отвечает одним пакетом META (см. §8.1).
- `cfg=2` в CMD — бэк отвечает серией пакетов с секциями (зоны, устройства, маппинг, пользователи) с пагинацией.

Фронт самостоятельно решает, когда запросить новый конфиг, сравнивая собственный сохранённый `cfgh` с тем, что лежит в META.

### 6.5 Алерты

Бэк отслеживает устройства, помеченные как алертные (binary_sensor с device_class door / motion / smoke / gas и аналогичные). Пока алерт активен, бэк отправляет PUSH с состоянием алерта **каждые 15 секунд**. Несколько активных алертов группируются в одну пачку (см. §7).

Когда алерт снимается, бэк отправляет финальный PUSH с новым состоянием и прекращает повторы.

### 6.6 Изменение конфигурации

Если в Home Assistant изменился состав устройств / зон / пользователей:

1. Бэк обнаруживает изменение через registry events HA.
2. Пересчитывает `cfgh`.
3. Рассылает broadcast пакет обновления конфига (дельта: список удалённых и добавленных устройств) либо новый META.
4. Фронты, услышавшие пакет, применяют дельту локально.
5. Фронт, пропустивший дельту (был оффлайн), при следующем PING запросит `cfg=2` целиком — расхождение `cfgh` это покажет.

## 7. Очередь сообщений

### 7.1 Принципы

- Очередь у каждой стороны (бэк и фронт) — независимая.
- **Ключ очереди исходящих:** `short_id` устройства (для CMD и PUSH по устройству). Для общих сообщений (CONFIG, STATUS-серия, алерты) — уникальный ключ сообщения.
- При повторной постановке сообщения с тем же ключом — старое затирается полностью, счётчик повторов сбрасывается.
- Приоритеты в очереди отсутствуют — все сообщения равноприоритетны.

### 7.2 Сортировка и обработка

- Каждое сообщение в очереди имеет поле `next_send_time` (точность мс).
- Очередь сортируется по `next_send_time` по возрастанию.
- Цикл обработки: взять верхний элемент → если `now >= next_send_time`, добавить в текущую пачку → обновить `next_send_time = now + период_повтора` → уменьшить `retries_left` → пересортировать → повторить.
- При `retries_left == 0` сообщение удаляется из очереди.
- **Первая отправка** сообщения происходит немедленно при постановке в очередь. Повторы — через `период_повтора` (настраивается, диапазон 5–60 секунд, по умолчанию 15).
- **Максимум повторов** — настраивается, по умолчанию 5.

### 7.3 Формирование пачки

- Лимит размера пакета — 200 байт.
- Процедура отправки берёт сообщения из очереди по порядку и пытается уместить в один пакет:
  - Если очередное сообщение влезает — добавляет.
  - Если не влезает — текущий пакет отправляется, для не уместившегося сообщения `next_send_time` обновляется, обработка переходит к следующему.
- Если текущий пакет пуст, а сообщение всё равно не влезает — оно фрагментируется (см. §7.4), и фрагменты отправляются подряд, пока сообщение не закончится. Только потом обработка переходит к следующим сообщениям.

### 7.4 Фрагментация

Любое сообщение, не помещающееся в один пакет, упаковывается во внешний конверт `tp=7` (FRAGMENT).

**Алгоритм отправителя:**
1. Сериализовать исходное сообщение в MessagePack — получить байты `B`.
2. Если `len(B) + накладные расходы <= 200` — отправить как есть, без фрагментации.
3. Иначе разрезать `B` на куски по `~180 байт` (с учётом конверта).
4. Для каждого куска отправить пакет:

```
{
  tp: 7,
  fid: <int, 16 бит, уникальный ID фрагментированного сообщения>,
  fi:  <int, индекс фрагмента, начиная с 0>,
  fn:  <int, всего фрагментов>,
  d:   <bytes, кусок исходных байт>
}
```

**Алгоритм приёмника:**
1. Получил пакет с `tp=7` → положил в буфер по ключу `fid`.
2. Когда в буфере собрано `fn` фрагментов — склеить по `fi`, распарсить исходное сообщение, передать в обычный обработчик.
3. Таймаут сборки — **60 секунд** с момента первого фрагмента. По таймауту буфер очищается.

**Ретраи фрагментов отдельно не предусмотрены.** Если фрагментированное сообщение потерялось — оно будет перезапрошено на верхнем уровне (например, фронт увидит расхождение `cfgh` и запросит `cfg=2` повторно).

### 7.5 Хранение обработанных сообщений

Подтверждённые / устаревшие сообщения не удаляются из памяти сразу — хранятся **15 минут** с момента последней активности. Это нужно для дедупликации: если по сети пришёл дубль уже обработанного сообщения, он отбрасывается без повторной обработки.

### 7.6 Поведение при отсутствии фронта

Если фронты считаются оффлайн (см. §6.1), бэк **не копит** очередь PUSH и алертов — это бессмысленно. Очередь остаётся пустой до возвращения фронта в онлайн (первый PING).

## 8. Конфигурация

### 8.1 META (cfg=1)

Один пакет:

```
{
  tp: 4,
  s: "meta",
  n: "Мой дом",
  gw: "!077ccb09",       # node ID шлюза
  ch: "LongFast",        # имя канала Meshtastic
  key: "AQ==",           # ключ канала (на MVP)
  upd: 60,               # период повторов очереди (сек)
  rty: 5,                # максимум повторов
  psh: 1,                # включена ли инициативная отправка PUSH
  hop: 0,                # hop limit
  cfgh: "3d9f21ac"       # хеш текущего конфига
}
```

### 8.2 Полный конфиг (cfg=2)

Несколько секций с пагинацией:

```
{tp:4, s:"ar",  pg:1, pgt:1, d:[...зоны]}
{tp:4, s:"dev", pg:1, pgt:3, d:[...устройства, по 5 в пакете]}
{tp:4, s:"mpg", pg:1, pgt:2, d:[...маппинг, по 8 в пакете]}
{tp:4, s:"usr", pg:1, pgt:1, d:[...пользователи]}
```

Пакеты ставятся в очередь и обрабатываются ею (см. §7). Отдельной паузы между пакетами CONFIG не задаётся — её обеспечивает общий период очереди.

### 8.3 Расчёт cfgh

`cfgh` считается **только по тому, что уходит на фронт** — то есть по экспортируемому виду конфига, а не по полному внутреннему состоянию бэка.

```
cfgh = MD5(json.dumps({
    "devices": [...export_devices()...],   # {short_id, n, t, a}
    "areas":   [...export_areas()...],     # {id, n}
    "users":   [...export_users()...]      # {id, hash}
}, sort_keys=True))[:8]
```

Хеш пересчитывается при каждом изменении состава экспортируемых данных. Изменения, видимые только бэку (например, смена `entity_id` при сохранении того же `short_id`), `cfgh` не меняют — фронту это знать не нужно.

## 10. Карточка Lovelace (card)

Карточка отображается на дашборде Home Assistant и выполняет три функции:

### 10.1 Шапка
Отображает:
- Имя дома (из META).
- Количество зарегистрированных фронтов.
- Количество онлайн-фронтов (тех, от кого PING поступал в последние 10 минут).

### 10.2 Поток пакетов
Скроллируемый список последних пакетов (входящих и исходящих).

Каждая строка содержит:
- Направление: входящий (↓) / исходящий (↑).
- Время (HH:MM:SS).
- Имя ноды-отправителя (`from` для входящих, шлюз — для исходящих).
- Тип пакета (CMD / PUSH / PING / ...).
- Вторая строка — минимальные данные пакета (`short_id` устройства, ключевые поля состояния).

При нажатии на строку — раскрытие в полные данные пакета (все поля в виде key: value).

Список хранится в памяти карточки, лимит — последние 200 пакетов.

### 10.3 Админ-раздел
Кнопка «Админ» открывает отдельную вкладку. Доступ — по вводу одного из паролей пользователя (механизм паролей определяется в §11).

В админ-вкладке:
- QR-код для онбординга нового фронта (содержит имя канала Meshtastic + PSK + node ID шлюза).
- Список зарегистрированных фронтов с возможностью отзыва.

---

## 11. Безопасность

- **PIN-коды и пароли** передаются хешами, не открытым текстом. Алгоритм — SHA-256, соль — общая для пары шлюз/приложение, задаётся при онбординге.
- **Шифрование канала** обеспечивается приватным каналом Meshtastic. На MVP допустимо использование публичного LongFast.
- **Онбординг нового фронта** (после MVP) — через QR-код, содержащий имя канала Meshtastic + PSK + node ID шлюза. На MVP конфиг копируется вручную.
- **Пароли пользователей и админ-доступ** — детально прорабатывается отдельно, см. открытые вопросы.

---

## 12. Логирование

Лог ведётся на всех трёх компонентах.

**Что пишется:**
- События очереди (постановка, отправка, повторы, удаление).
- Исполнение команд (получение CMD, вызов сервиса HA, отправка PUSH).
- Изменения состояния, прилетевшие от HA.
- Краши и исключения.

**Уровни:** DEBUG / INFO / WARN / ERROR.

**Ротация:**
- back — файлы, ротация по размеру (5 × 10 МБ).
- app — кольцевой буфер в памяти + экспорт в файл по запросу пользователя.
- card — буфер в памяти браузера, видим в DevTools.

---

## 13. Несколько фронтов

Поддерживается одновременная работа нескольких экземпляров приложения с одним шлюзом.

- Все на одном приватном канале Meshtastic — слышат друг друга.
- Команды от любого фронта обрабатываются бэком.
- PUSH с новым состоянием идёт broadcast — применяется всеми фронтами.
- Одновременные команды от разных фронтов на одно устройство выполняются в порядке прихода. Последняя выигрывает. Блокировок нет.
- При `req="all"` от одного фронта остальные фронты получают STATUS-серию «попутно» — это ожидаемо.

## 14. Открытые вопросы (для следующих итераций)

- Возврат CONFIRM в случаях, где гарантия доставки критична (alarm arming, lock).
- Алгоритм определения «алертных» устройств — фиксированный список device_class или конфигурируемый.
- Поведение back при недоступности самого Home Assistant (рестарт, тормоза) — буферизация команд или быстрый отказ.
- Шифрование payload поверх Meshtastic (дополнительный слой к ключу канала).
- Схема паролей пользователей и админ-доступа в card.
