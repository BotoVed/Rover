# Rover — Спецификация протокола и системы

**Версия:** MVP 0.1
**Статус:** черновик после согласования

---

## 1. Назначение системы

Rover (**R**emote **O**ver **R**adio) — система дистанционного управления умным домом Home Assistant в условиях отсутствия интернета или его блокировки. Целевая среда — дом на колёсах (RV), яхта, дача. Связь между мобильным приложением и шлюзом обеспечивается радиосетью LoRa через стек Meshtastic.

## 2. Архитектура

Система состоит из трёх компонентов:

- **back** — кастомная интеграция Home Assistant (Python).
- **app** — мобильное приложение под Android (Kotlin).
- **card** — карточка Lovelace для дашборда Home Assistant (TypeScript + LitElement).

Допускается одновременная работа нескольких экземпляров приложения с одним шлюзом.

## 3. Транспортный слой

### 3.1 Радиосеть

- **Стек:** Meshtastic.
- **Канал:** приватный канал Meshtastic с заданным именем и PSK. На MVP допускается LongFast / `AQ==`.
- **Порт приложения:** `PRIVATE_APP = 256`.
- **Hop limit:** настраивается, по умолчанию `0`.
- **Адресация:** все пакеты broadcast (`to = 0xFFFFFFFF`).
- **Подтверждение доставки:** все пакеты отправляются с `want_ack=true`. Meshtastic возвращает implicit ACK при ретрансляции пакета на канале или NAK при отсутствии ответа за время firmware-таймаута. ACK-механизм используется в очереди для раннего завершения retry-цикла (см. §7.7).

### 3.2 Сериализация

- **Формат:** MessagePack (бинарный).
- **Лимит размера пакета:** 200 байт payload.
- **Обязательное поле:** `tp` (Int) — тип пакета.

### 3.3 Совместимость числовых типов

Числа могут прийти как Int / Long / UInt. Приёмники обязаны принимать оба варианта.

## 4. Типы пакетов

| tp | Направление | Название | Описание |
|----|-------------|----------|----------|
| 2 | HA → App | STATUS | Состояние устройства в ответ на запрос |
| 3 | HA → App | PUSH | Изменение состояния (инициативное или после команды) |
| 4 | HA → App | CONFIG | Секция конфигурации (страница с pg/pgt) |
| 5 | App → HA | CMD | Команда устройству или запрос состояния |
| 6 | App ↔ HA | PING / PONG | Keepalive с обменом хешами секций конфига |
| 7 | Любое | FRAGMENT | Часть фрагментированного сообщения |
| 8 | App → HA | REQ | Запрос секции конфига (или одной её страницы) |

**Тип `tp=1` (CONFIRM) изъят.** Подтверждение исполнения команды — это broadcast PUSH с новым состоянием.

## 5. Идентификация устройств и хранение конфига

### 5.1 Источник истины

Все данные конфигурации хранятся **на бэке**. Фронт долговременно хранит только параметры подключения и кеш последнего полученного конфига для оффлайн-отображения.

### 5.2 Хеш устройства

`short_id` — **2-байтовый Int** (16 бит), первые 2 байта MD5 от `entity_id`. При коллизии — пересчёт с инкрементом солта. Назначенный однажды `short_id` за устройством закреплён навсегда.

### 5.3 Что хранит бэк

```
short_id → {
    entity_id, t, n, a, u
}
```

Плюс `areas` и `users`. Полные записи остаются на бэке.

### 5.4 Что уходит на фронт

- **devices** — `{short_id, n, t, a}` (без `entity_id`).
- **areas** — `{id, n}`.
- **users** — `{id, hash}` для локальной авторизации.

В обычных пакетах (CMD, PUSH, STATUS) передаётся только `short_id`.

### 5.5 Типы устройств

| Код | HA domain | Команды | Поля состояния |
|-----|-----------|---------|----------------|
| L | light | s, bri, ct | s, bri, ct |
| SW | switch | s | s |
| C | climate | s, th, md, fn | s, th, tc, md, fn |
| WH | water_heater | s, th | s, th, tc, md |
| F | fan | s, sp | s, sp |
| CV | cover | cmd (open/close/stop), pos | st, pos |
| LK | lock | cmd (lock/unlock) | st |
| BS | binary_sensor | — | s |
| S | sensor | — | v (string), u |
| SI | siren | s | s |
| B | button / scene | cmd (press) | — |
| A | alarm_control_panel | cmd (arm_home/arm_away/arm_night/disarm), pin | st |
| H | humidifier | s, th | s, th, tc |

**Примечание про `bri` (яркость light):** диапазон `0..100` (проценты). Бэк конвертирует в HA brightness `0..255`.

**Примечание про `th` для humidifier:** для типа `H` поле `th` означает целевую влажность.

**Примечание про `v` (sensor value):** передаётся **строкой**, не Double. Покрывает любые HA-сенсоры (числа, enum, timestamp).

## 6. Семантика обмена

### 6.1 PING / PONG — обмен версиями конфига

Фронт отправляет **PING** раз в 2 минуты с мапой своих хешей секций конфига:

```
{tp:6, ts:1717145000, h:{m:"ab12", u:"cd34", a:"ef56", d:"78ab"}}
```

Если фронт ещё ничего не получал — отсутствующие или пустые хеши.

Бэк на каждый PING отвечает **PONG** с мапой своих текущих хешей:

```
{tp:6, ts:1717145000, h:{m:"ab12", u:"cd34", a:"ef56", d:"9911"}}
```

`ts` в PONG — эхо `ts` из PING (для удобства корреляции). В инициативном PONG (см. §6.6) — текущее время бэка.

**Бэк никаких решений по содержимому хешей не принимает.** Это чистый обмен версиями.

**Фронт** при получении PONG сравнивает хеши секций и при расхождении шлёт REQ (см. §6.4) на нужные секции.

Если бэк не получает PING более **10 минут** — фронт считается оффлайн.

### 6.2 Жизненный цикл команды

1. Пользователь меняет значение на фронте → UI «серый» (неподтверждённый).
2. Фронт ставит CMD в очередь, ключ = `short_id`.
3. Бэк получает CMD → вызывает HA-сервис → ждёт 500 мс → читает state → шлёт broadcast PUSH.
4. Все фронты получают PUSH. Инициатор снимает «серый», остальные просто обновляют.
5. Если PUSH не пришёл за `период × попыток` — устройство остаётся серым. Ошибка не показывается.

### 6.3 Запрос актуальных состояний

В CMD:
- `req="all"` — бэк отвечает серией STATUS-пакетов по одному на устройство.
- `req=<short_id>` — бэк отвечает одним STATUS.

### 6.4 Запрос конфигурации — пакет REQ (tp=8)

Фронт точечно запрашивает секцию (или её страницу) пакетом REQ:

```
{tp:8, s:"d"}             # вся секция devices, все страницы
{tp:8, s:"d", pg:3}       # только страница 3 секции devices
{tp:8, s:"m"}             # секция meta (всегда 1 страница)
{tp:8, s:"u"}             # секция users
{tp:8, s:"a"}             # секция areas
```

Бэк по REQ кладёт запрошенные пакеты в очередь.

**Поле `cfg` в CMD удалено.** Раньше `cfg=1` запрашивало META, `cfg=2` весь конфиг — теперь это всё через REQ.

### 6.5 Инициативный PONG при изменении хешей

Бэк хранит свои текущие хеши секций. При любом событии HA, меняющем экспортируемые данные (`state_changed` для зарегистрированных устройств, изменение areas, изменение users):

1. Бэк пересчитывает хеши секций.
2. Если хоть один хеш изменился — обновляет сохранённые и **сразу** ставит PONG в очередь broadcast'ом.
3. Все фронты получают PONG → сравнивают со своими хешами → шлют REQ на изменившиеся секции.

Инициативный PONG ставится в очередь по ключу `"pong"` — то есть если хеши изменились второй раз до отправки, в очереди окажется только последняя версия (затирание по ключу).

Поле `ts` в инициативном PONG — текущее время бэка (нет PING, не от чего эхо).

## 7. Очередь сообщений

### 7.1 Принципы

- Очередь у каждой стороны независимая.
- **Ключ очереди:** `short_id` устройства (для CMD/PUSH/STATUS по устройству), `"pong"` для PONG, уникальные ключи для CONFIG-страниц.
- При повторной постановке с тем же ключом — старое затирается, счётчик повторов сбрасывается. Затирание работает в любой фазе элемента (active, awaiting_ack, suspended).
- Приоритеты отсутствуют — все пакеты обрабатываются одинаково.
- Все пакеты отправляются с `want_ack=true` — нет разделения на «критичные» и «не критичные».

### 7.2 Фазы элемента очереди

| Фаза | Описание |
|------|----------|
| `active` | Готов к отправке или ожидает следующего цикла. `next_send_time` определяет момент. |
| `awaiting_ack` | Отправлен, ждёт ACK от Meshtastic. `sent_at` зафиксирован, `packet_id` известен. |
| `suspended` | Все попытки исчерпаны без ACK. Лежит без отправок до пробуждения. |

### 7.3 Сортировка и обработка

- Элементы в фазе `active` сортируются по `next_send_time`.
- Цикл обработки берёт только `active` с `next_send_time ≤ now`.
- Элементы в `awaiting_ack` и `suspended` не участвуют в формировании пачки.
- Первая отправка — немедленно (`next_send_time = now` при `put`).
- Повторы — каждые `queue_period` (настройка, дефолт 15 сек).
- Максимум повторов — `max_retries` (настройка, дефолт 5).
- ACK-таймаут — `ack_timeout` (настройка, дефолт 10 сек). Должен быть меньше `queue_period`.

### 7.4 Формирование пачки

Без изменений — Batcher собирает в пачку сообщения в фазе `active`, лимит 200 байт. Не помещающиеся пропускаются. Не влезающие в пустую пачку — фрагментируются через `tp=7`.

### 7.5 Фрагментация (tp=7)

Без изменений — конверт `{tp:7, fid, fi, fn, d}`, сборка по `fid`, таймаут 60 сек. **Каждый фрагмент тоже отправляется с want_ack=true** — потеря одного фрагмента означает потерю всего сообщения, обнаружение через NAK ускоряет retry.

### 7.6 Хранение обработанных сообщений

Без изменений — обработанные входящие хранятся 15 минут для дедупликации.

### 7.7 ACK-механизм и жизненный цикл

**Отправка:**
1. Item в фазе `active`, `next_send_time ≤ now` → попадает в пачку.
2. Пачка отправлена в Meshtastic с `want_ack=true`.
3. Item переходит в `awaiting_ack`, `sent_at = now`, `packet_id` запомнен.
4. `retries_left -= 1`.

**Возможные исходы:**

| Событие | Действие |
|---------|----------|
| Пришёл ACK с `packet_id` | Item удаляется из очереди. Фронт-отправитель помечается онлайн. |
| Пришёл NAK с `packet_id` | Если `retries_left > 0` → `active`, `next_send_time = now + queue_period`. Иначе → `suspended`. |
| `now - sent_at > ack_timeout` | То же, что NAK: retry или suspended. |
| `packet_id` неизвестен (ACK на удалённый/затёртый item) | Молчаливо игнорируется. |

**Перезапись item в `awaiting_ack`:**
Если по тому же ключу пришёл новый `put`, старый item затирается. Будущий ACK или NAK на его `packet_id` игнорируется (см. выше). Новый item начинает с `retries_left = max_retries`, фаза `active`, `next_send_time = now`.

### 7.8 Suspended и пробуждение

Когда у item исчерпаны все retries без ACK — он не удаляется, а переходит в `suspended`. Это сохраняет последнее намерение «доставить» на случай возврата фронта в зону связи.

**Пробуждение всех suspended** происходит, когда бэк понимает, что фронт онлайн:
- Получен PING от фронта, ИЛИ
- Получен ACK на любой другой отправленный пакет (это значит, кто-то в эфире нас услышал).

При пробуждении:
- Все элементы в фазе `suspended` переводятся в `active`.
- `retries_left = max_retries`.
- `next_send_time = now`.
- В следующем цикле они улетают пачкой по обычным правилам Batcher.

**Рост suspended ограничен.** Перезапись по ключу затирает старые suspended-элементы новыми событиями. Максимум suspended-элементов ≈ числу уникальных ключей (устройств + системных пакетов), то есть десятки, не тысячи.

### 7.9 Поведение при отсутствии фронта

Если PING-ов от фронта не было дольше 10 минут и ни один ACK не приходил — фронт считается оффлайн. Все элементы в `awaiting_ack` со временем уйдут в `suspended` через retries. Новые `put` создают `active`-элементы, которые после серии неудачных попыток тоже уйдут в `suspended`. Эфир молчит. Очередь не растёт за счёт затирания по ключу.

При появлении любого признака онлайна (PING или ACK) — массовое пробуждение, отправка накопленного.

## 8. Конфигурация

### 8.1 Секции конфига и их хеши

Конфиг логически разделён на 4 секции с независимыми хешами:

| Код | Секция | Источник данных |
|-----|--------|-----------------|
| `m` | meta | config_entry (имя дома, канал, PSK, hop, period, retries, push_enabled) + node ID шлюза |
| `u` | users | Registry.export_users() |
| `a` | areas | Registry.export_areas() |
| `d` | devices | Registry.export_devices() |

Хеш каждой секции — **MD5[:4]** (4 hex знака) от sort_keys JSON-сериализации содержимого секции.

Длина 4 знака (16 бит) достаточна: задача хеша — сигнал «изменилось/не изменилось», а не уникальность. Случайная коллизия раз в ~65000 изменений приемлема.

### 8.2 Секция meta (`m`)

Одна страница:

```
{
  tp:4, s:"m", pg:1, pgt:1,
  d:{
    n:"Мой дом",          # имя дома (из config_entry)
    gw:"!077ccb09",        # node ID шлюза
    ch:"LongFast",         # имя канала Meshtastic
    key:"AQ==",            # ключ канала
    upd:15,                # период повторов очереди (сек)
    rty:5,                 # максимум повторов
    psh:1,                 # включена ли инициативная отправка PUSH
    hop:0,                 # hop limit
    salt:"<32-hex случайная соль>"  # общая соль для хеширования паролей (см. §11)
  }
}
```

### 8.3 Секция users (`u`)

```
{tp:4, s:"u", pg:1, pgt:1, d:[
  {id:"admin", hash:"<SHA-256 пароля с солью>"},
  {id:"guest", hash:"..."}
]}
```

### 8.4 Секция areas (`a`)

```
{tp:4, s:"a", pg:1, pgt:1, d:[
  {id:"salon", n:"Гостиная"},
  {id:"kitchen", n:"Кухня"}
]}
```

### 8.5 Секция devices (`d`)

Разбивается на страницы по 5 устройств:

```
{tp:4, s:"d", pg:1, pgt:6, d:[...5 devices...]}
{tp:4, s:"d", pg:2, pgt:6, d:[...5 devices...]}
...
{tp:4, s:"d", pg:6, pgt:6, d:[...последние devices...]}
```

Каждая страница — самодостаточная. Если фронт получил pg=1,2,4,5,6, но pg=3 потеряна — он шлёт `{tp:8, s:"d", pg:3}` и получает только её. Не нужно перезапрашивать всю секцию.

Если в одну страницу не влезает её содержимое (например, длинные имена) — Batcher автофрагментирует страницу через tp=7. Это редкий fallback-механизм.

### 8.6 Когда фронт обновляет свой локальный хеш секции

Локальный хеш секции у фронта обновляется **только когда вся секция полностью собрана**. Если пришли 3 из 5 страниц devices — хеш `d` всё ещё старый, фронт продолжает запрашивать недостающие страницы через REQ.

## 9. Поля пакетов

### 9.1 CMD (App → HA, tp=5)

| Поле | Тип | Описание |
|------|-----|----------|
| tp | Int | Всегда 5 |
| id | Int | `short_id` устройства (для команды на устройство) |
| cmid | Int | Уникальный ID команды (для дедупа) |
| s | Int | Состояние 0/1 |
| bri | Int | Яркость 0..100 |
| ct | Int | Цветовая температура (mireds) |
| th | Double | Целевая температура или влажность |
| md | String | Режим (cool/heat/auto/...) |
| fn | String | Скорость вентилятора |
| sp | Int | Скорость % |
| pos | Int | Позиция жалюзи % |
| cmd | String | Команда (open/close/lock/unlock/press/arm_*/disarm) |
| pin | String | PIN для сигнализации (хеш) |
| req | Any | `"all"` или `short_id` — запрос состояний |
| hl | Int | Hop limit |
| ts | Long | Unix timestamp отправителя |

**Поле `cfg` удалено** — запрос конфигурации идёт через tp=8 REQ.

### 9.2 STATUS / PUSH (HA → App, tp=2 / tp=3)

| Поле | Тип | Когда используется |
|------|-----|--------------------|
| tp | Int | Всегда |
| id | Int | Всегда — `short_id` устройства |
| s | Int | light / switch / fan / siren / humidifier / binary_sensor |
| bri | Int | light (0..100) |
| ct | Int | light |
| th | Double | climate / water_heater / humidifier (целевая) |
| tc | Double | climate / water_heater / humidifier (текущая) |
| md | String | climate / water_heater |
| fn | String | climate |
| sp | Int | fan |
| st | String | cover / lock / alarm |
| pos | Int | cover |
| v | String | sensor (значение любого вида: число, enum, timestamp) |
| u | String | sensor (единица измерения) |

### 9.3 PING / PONG (tp=6)

| Поле | Тип | Описание |
|------|-----|----------|
| tp | Int | 6 |
| ts | Long | Unix timestamp. В PING — отправителя. В PONG ответном — эхо. В PONG инициативном — текущее время бэка. |
| h | Map | Мапа `{section_code: hash4hex}`. В PING — хеши фронта. В PONG — хеши бэка. |

**Примечание о want_ack:** PING со стороны фронта отправляется с `want_ack=true`. Это даёт фронту раннее подтверждение, что шлюз в зоне связи (без ожидания PONG). PONG со стороны бэка отправляется аналогично — с `want_ack=true`, как и все остальные пакеты (см. §7.1).

### 9.4 FRAGMENT (tp=7)

| Поле | Тип | Описание |
|------|-----|----------|
| tp | Int | 7 |
| fid | Int | ID фрагментированного сообщения |
| fi | Int | Индекс фрагмента, с 0 |
| fn | Int | Всего фрагментов |
| d | Bytes | Кусок исходных байт |

### 9.5 REQ (App → HA, tp=8)

| Поле | Тип | Описание |
|------|-----|----------|
| tp | Int | 8 |
| s | String | Код секции: `"m"`, `"u"`, `"a"`, `"d"` |
| pg | Int | (опционально) Номер запрашиваемой страницы. Если опущен — вся секция целиком. |

### 9.6 CONFIG (HA → App, tp=4)

| Поле | Тип | Описание |
|------|-----|----------|
| tp | Int | 4 |
| s | String | Код секции (`"m"`, `"u"`, `"a"`, `"d"`) |
| pg | Int | Номер страницы (с 1) |
| pgt | Int | Всего страниц в секции |
| d | Any | Содержимое страницы (для `m` — dict, для `u/a/d` — list of dicts) |

## 10. Карточка Lovelace (card)

### 10.1 Шапка
- Имя дома (из секции meta).
- Количество зарегистрированных фронтов.
- Количество онлайн-фронтов.

### 10.2 Поток пакетов
Скроллируемый список последних 200 пакетов (входящих и исходящих). При клике — раскрытие в детали.

### 10.3 Админ-раздел
Доступ по паролю. Содержит QR онбординга нового фронта и список зарегистрированных фронтов.

---

## 11. Безопасность

- PIN-коды и пароли передаются хешами (SHA-256 с солью).
- Шифрование канала — приватный канал Meshtastic.
- Онбординг через QR — после MVP. На MVP — ручной ввод.
- **Общая соль для хеширования паролей пользователей** генерируется один раз при создании ConfigEntry интеграции, хранится в `entry.data` и передаётся фронту в составе секции `m` (meta). Это позволяет фронту локально хешировать вводимый пользователем пароль и сравнивать с хешами из секции `u` (users).

---

## 12. Логирование

Логи ведутся на back (файлы, ротация), app (кольцевой буфер + экспорт), card (буфер в браузере + DevTools).

---

## 13. Несколько фронтов

Поддерживается. Все на одном приватном канале, broadcast доходит до всех. Бэк не различает фронты по node ID.

---

## 14. Открытые вопросы

- CONFIRM для критичных команд (alarm/lock).
- Поведение back при недоступности HA (буферизация / отказ).
- Шифрование payload поверх Meshtastic.
- Схема паролей пользователей и админ-доступа в card.
